## Findings

#### Finding 1: Compaction with structured preservation (Anthropic context engineering)
**Source:** https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
**What it is:** Anthropic's engineering team defines "compaction" as summarizing a context nearing its limit and reinitializing with the summary, with a specific prescription: distill while preserving architectural decisions and unresolved issues, discard redundant tool outputs; clear tool call results once processed as the lowest-risk first step.
**Already in shmorch?** Partially — Shmorch has "compress proactively" and compaction steps in context.md, but lacks the specific discard-ordering rule (tool outputs first, reasoning last) and the explicit preservation checklist (architectural decisions + unresolved issues only).
**Applicability:** High — maps directly onto context.md's compaction steps and would make them more precise.

#### Finding 2: 40% context utilization ceiling / tool loadout ceiling
**Source:** https://blog.logrocket.com/llm-context-problem-strategies-2026/
**What it is:** Practitioners have converged on keeping active context under ~40% utilization; tool loadouts above ~30 tools cause selection failure; dynamic tool selection (choosing only relevant tools per task) improved performance 44% and execution speed 77%.
**Already in shmorch?** No — Shmorch has no quantitative thresholds for context or tool loadout. The "cost discipline" rules are qualitative.
**Applicability:** Med — Shmorch doesn't manage tool loadout programmatically, but the 40% utilization threshold is a useful heuristic for knowing when to proactively compact; worth encoding as a trigger.

#### Finding 3: Just-in-time context loading vs. pre-loading
**Source:** https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
**What it is:** Rather than front-loading all relevant files at session start, maintain lightweight references (file paths, queries) and retrieve dynamically during execution — progressive disclosure rather than full upfront scan.
**Already in shmorch?** Partially — Session Start Checklist reads context.md, session.md, plan.md upfront (3 files), which is bounded. Build workflow passes stack.md to implementers. No explicit instruction against broader upfront scans.
**Applicability:** High — The "avoid broad scans unless requested" cost discipline rule is related but not the same thing; the JIT pattern explicitly applies to subagent task prompts (what do you inject vs. what do you reference).

#### Finding 4: Harness engineering — deterministic constraint enforcement vs. probabilistic prompt-based compliance
**Source:** https://www.augmentcode.com/guides/harness-engineering-ai-coding-agents
**What it is:** Harness engineering separates "telling an agent to follow standards" (probabilistic, drifts) from "wiring a linter that blocks the PR" (deterministic, enforced). Three-layer model: Constraint Harnesses (feedforward rules files), Feedback Loops (error messages that include remediation instructions for self-correction), Quality Gates (CI blocks).
**Already in shmorch?** Partially — Shmorch has stack.md and "never do without asking" rules encoded in CLAUDE.md, and the build workflow's DoD checklist. No explicit instruction to make constraint violations self-correcting (i.e., that error messages from linters should be treated as self-correction prompts rather than human escalations).
**Applicability:** High — The gap between Shmorch's current soft-constraint model (instructional, in markdown) and harness engineering's hard-constraint model (linter gates in CI) is real. Shmorch can't wire CI itself, but it can instruct implementers to ensure linter errors include remediation text and to never suppress lint violations inline.

#### Finding 5: AGENTS.md as cross-session single source of truth for constraints
**Source:** https://www.builder.io/blog/agents-md
**What it is:** AGENTS.md is a standardized (August 2025, multi-vendor) root file that functions as a README for AI agents. It encodes: version specificity, stylistic exemplars (point to actual files), permission matrices (what can run autonomously vs. requires approval), and prohibitions. Nested AGENTS.md files in subdirectories handle per-area overrides. Multiple tools (Cursor, Codex, Claude Code, etc.) now read it natively.
**Already in shmorch?** Partially — Shmorch uses CLAUDE.md for constraint encoding, which Claude Code reads natively. AGENTS.md would add cross-tool portability and the nested/hierarchical pattern. The permission matrix pattern (auto-allowed vs. ask-first) is absent from current Shmorch templates.
**Applicability:** High — The permission matrix and file-exemplar patterns are directly applicable to Shmorch's init template and could strengthen constraint propagation without infrastructure changes.

#### Finding 6: Plan-Execute-Verify (PEV) phase gates for constraint propagation in multi-agent workflows
**Source:** https://www.augmentcode.com/guides/harness-engineering-ai-coding-agents
**What it is:** PEV separates planning from execution with explicit phase gates: Plan phase decomposes with acceptance criteria; Execution is bounded by the plan with pre-execution gates that validate tool calls and workspace boundaries before execution begins; Verify checks plan alignment (not just test pass/fail — "did the agent use the right middleware?"). This catches constraint violations that test suites miss.
**Already in shmorch?** Partially — Shmorch's build workflow has spec → design → implement → DoD, which is similar. Missing: pre-execution gates that validate tool calls and workspace boundaries before an implementer subagent begins, and a plan-alignment verification step that goes beyond "tests pass."
**Applicability:** High — Directly applicable to Step 2 (Decompose) in build.md; a pre-execution gate between Step 2 and Step 3b would catch out-of-scope actions before they happen.

#### Finding 7: Coordinator-Implementor-Verifier architecture with dependency graph propagation
**Source:** https://www.augmentcode.com/guides/spec-driven-ai-code-generation-with-multi-agent-systems
**What it is:** When a spec or API contract changes, a Coordinator agent identifies all affected services/files, an Implementor executes changes in isolated worktrees, and a Verifier checks implementations against living specs — not just tests. The key: dependency graph traversal happens before implementation, not after.
**Already in shmorch?** Partially — Shmorch has graph thinking (trace lateral implications) and bidirectional link maintenance. Missing: a pre-implementation propagation step that explicitly enumerates affected docs/tests/services before the build starts, and a post-implementation verification step that checks doc-code alignment separately from test passage.
**Applicability:** High — Maps onto the build workflow's Step 2 (Decompose) and Step 4 (DoD). The gap is that Shmorch's DoD checklist asks "does docs need updating?" reactively, after code is written. A propagation step before Step 3 would flip this.

#### Finding 8: Amazon Kiro bidirectional spec-code synchronization
**Source:** https://www.augmentcode.com/guides/spec-driven-ai-code-generation-with-multi-agent-systems (citing Kiro)
**What it is:** Kiro's approach makes spec-code sync bidirectional: developers can update code and request spec updates, or update specs to trigger implementation tasks. Changes in either direction are tracked and flagged.
**Already in shmorch?** No — Shmorch's flow is spec → code, one direction. There is no pattern for "code changed, now update the spec/docs." The vacuum workflow touches cleanup but not bidirectional sync.
**Applicability:** Med — Shmorch already has "continuous state updates" and "documents stay clean," which handle the intent partially. Full bidirectionality requires a trigger when code changes happen outside a planned build task (e.g., hotfixes, refactors) — currently those paths don't explicitly prompt doc-sync.

## Proposals

#### Proposal 1: Add discard-ordering rule to compaction steps in context.md
**Based on:** Finding 1
**File to change:** `~/.claude/skills/shmorch/workflows/context.md`
**Proposed change:** In the "Compaction steps" list, add a step 0 before the existing steps:
```
0. Clear processed tool outputs first — these are the highest-volume, lowest-value tokens. 
   Do not summarize them; discard entirely.
```
And add to step 2 (write decisions): "Preserve architectural decisions and unresolved issues verbatim — these must survive compaction with no information loss."
**Expected improvement:** Compaction becomes systematic rather than judgment-based; implementers know exactly what to cut first and what to never cut.

#### Proposal 2: Add 40% utilization trigger to context.md
**Based on:** Finding 2
**File to change:** `~/.claude/skills/shmorch/workflows/context.md`
**Proposed change:** Add a section "Compaction Triggers" at the top, before "Compression: When a Thread Gets Too Long":
```markdown
## Compaction Triggers

Compact when any of the following is true:
- Context window estimated above 40% utilization
- Thread has exceeded ~20 exchanges without a clean stopping point
- Tool outputs accumulating without being referenced again
- User-visible symptoms: responses slowing, repeating context, losing track of prior decisions
```
**Expected improvement:** Removes guesswork about when to compact; gives the agent a quantitative signal.

#### Proposal 4: Add permission matrix to init template (CLAUDE.md / AGENTS.md pattern)
**Based on:** Finding 5
**File to change:** `~/.claude/skills/shmorch/commands/init.md` (or the CLAUDE.md template generated by init)

#### Proposal 5: Add pre-execution propagation gate to build.md Step 2
**Based on:** Findings 6 and 7
**File to change:** `~/.claude/skills/shmorch/workflows/build.md`
**Proposed change:** After Step 2 (Decompose) and before Step 3a/3b, add a required gate:

```markdown
## Step 2b — Propagation gate (required before any implementation)

Before writing code, enumerate:
1. **Affected docs:** Which architecture docs, API docs, or track docs will need updating if this spec is implemented? List them.
2. **Affected tests:** Which existing tests are likely to need updating? List them. If none, confirm why.
3. **Affected decisions:** Does this implementation require a new entry in decisions.md, or update an existing one?

Write this list to `docs/state/plan.md` under the active task.

**Gate condition:** Do not start Step 3 until this list exists. When implementation completes (Step 4), use this list as the doc/test update checklist — not just the DoD checklist.
```
**Expected improvement:** Catches doc-code drift before it happens rather than catching it at DoD; flips the doc-update step from reactive to proactive; reduces the "I forgot to update the architecture doc" failure mode.

#### Proposal 6: Add post-implementation plan-alignment check to DoD in build.md
**Based on:** Finding 6 (PEV Verify phase)
**File to change:** `~/.claude/skills/shmorch/workflows/build.md`
**Proposed change:** Add to Step 4 (Definition of Done), under a new heading "### Plan alignment":
```markdown
### Plan alignment
- Does the implementation use the patterns and dependencies specified in the design? (Not just "tests pass" — actively verify no deprecated API, no new undeclared dependency, no undeclared framework was introduced.)

```
**Expected improvement:** Catches the class of constraint violations that pass tests but violate architecture (e.g., agent reaches for a deprecated library that still works, silently).

#### Proposal 7: Add bidirectional sync trigger for out-of-plan code changes
**Based on:** Finding 8
**File to change:** `~/.claude/skills/shmorch/workflows/vacuum.md` (or shmorch-core.md safety rules)
**Proposed change:** Add to the Safety Rules section in shmorch-core.md, under "Tests and Docs Are the Source of Truth":
```markdown
**Bidirectional sync rule:** When code changes happen outside a planned build task (hotfixes, refactors, quick edits), treat them as spec-first violations and immediately flag:
```

**Expected improvement:** Closes the gap where quick fixes accumulate doc drift; makes bidirectional sync a reflex not just a build-time practice.

## Deferred

**Dynamic tool loadout selection (Finding 2 — 44% performance gain from relevant-tool-only loadout):** The 44% gain from dynamic tool selection is compelling, but Shmorch doesn't control which tools are available to subagents — that's set by Claude Code's harness. Relevant when Shmorch gains the ability to configure per-agent tool sets (e.g., via MCP tool filtering or agent team configurations).

**CI quality gates (Finding 4 — harness engineering, Layer 3):** Blocking CI pipelines with lint gates require CI infrastructure that may not exist in proof-sprint or R&D stages. Becomes relevant at `productionization` stage. The project stage table in shmorch-core.md could note "CI constraint gates recommended" at productionization stage.

**Coordinator-Implementor-Verifier as a named agent team pattern (Finding 7):** Shmorch's build workflow already has a similar shape, but formalizing it as a named, reusable team pattern requires the agent teams infrastructure to be stable. Defer until `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` is out of experimental.

**A2RAG adaptive graph-RAG retrieval:** Requires external vector store infrastructure and semantic graph indexing — out of scope for a skill-level orchestrator. Relevant if Shmorch ever integrates with an external memory MCP.
